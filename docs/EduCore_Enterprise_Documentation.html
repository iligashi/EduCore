Project Charter / Vision Document

ItemProject Definition
Project NameEduCore LMS & DMS
Project DescriptionEduCore is a full-stack platform that combines learning management and digital management operations for schools, bootcamps, training providers, or university departments. It provides role-based portals, course/class management, admissions, documents, lessons, assignments, attendance, gradebook, certificates, quizzes, notifications, search, reporting, import/export, and an AI-style Copilot workspace.
Business GoalsCentralize educational operations, reduce manual administration, improve instructor and learner visibility, support digital admissions and document review, and provide reporting for academic decision making.
Problem StatementEducational teams often manage applications, classes, attendance, assignments, lesson content, documents, communications, and reports in disconnected tools. This causes duplicate work, weak visibility, slow student support, and inconsistent records. EduCore addresses this by combining operational records and learning workflows in one portal.
Target UsersAdministrators, instructors, students, public applicants, academic leadership, admissions staff, IT/deployment maintainers, and project evaluators.
StakeholdersProject sponsor, product owner, academic administration, instructors, learners, admissions/review team, support staff, developers, database administrator, DevOps maintainer, security reviewer.

KPITargetMeasurement Method
Role coverageAdmin, instructor, student, and public applicant flows complete for MVP.Acceptance test checklist and demo scripts.
Administrative efficiencyReduce manual student/course/attendance/report work by at least 40 percent.Compare manual workflow steps before and after adoption.

Scope

  • JWT authentication and role-based authorization.
  • Admin operations for users, students, instructors, applications, courses, classes, enrollments, documents, CMS, reports, certificates, and import/export.
  • Instructor operations for class content, assignments, grading, attendance, quizzes, success center, notifications, and class analytics.
  • Student operations for dashboard, lessons, assignments, submissions, documents, gradebook/certificates, quizzes, notifications, recommendations, and Copilot.
  • Public course catalog and course application submission.
  • MySQL plus MongoDB persistence, Socket.IO realtime events, and Docker Compose local deployment.

Out Of Scope

  • Payment processing, invoicing, payroll, and finance modules.
  • Native iOS/Android apps.
  • External SIS/LTI integrations, enterprise SSO, and SCORM/xAPI runtime.
  • Production queue workers, Redis cache, CDN, object storage migration, and full-text search engine.
  • Advanced proctoring, plagiarism detection, and video conferencing.
  • Multi-tenant institution isolation unless added as a future phase.

Budget

The repository does not define a formal monetary budget. Recommended planning assumption: treat the current version as an academic or MVP project with internal labor and low-cost local infrastructure. A production pilot should budget for hosting, managed MySQL/MongoDB or VM capacity, object storage, domain/TLS, email delivery, monitoring, backups, and security review. Expected MVP infrastructure range is approximately USD 50-300/month depending on managed services and traffic.

Timeline

The repository includes a six-sprint plan: setup, authentication, DMS, LMS, advanced features, and finalization. A realistic capstone/MVP timeline is 6-10 weeks with one sprint per week or per two weeks. Production hardening should be planned as a separate phase after MVP acceptance.

Business Requirements Document (BRD)

Executive Summary

EduCore is designed to replace fragmented school operations with one authenticated portal. It serves administrators who need operational control, instructors who need class delivery tools, students who need learning and communication workflows, and applicants who need a public application path. The project already implements a broad MVP surface, but production readiness requires stronger automated tests, secret hygiene, CI/CD, monitoring, and backup procedures.

Business Objectives

ObjectiveBusiness RationaleImplemented Evidence
Increase operational visibilityAdmins need dashboards, reports, and searchable records.Reports routes, dashboard page, search page, success center.
Reduce manual workManual spreadsheets and disconnected records cause mistakes.Student/instructor CRUD, enrollment, attendance bulk save, import/export.
Improve student experienceStudents need one place for classes, work, grades, documents, quizzes, and notifications.Student home, lessons, work, gradebook, documents, quiz and inbox flows.
Improve instructor workflowInstructors need content, attendance, grading, quiz, and intervention tools.Instructor home, Class Studio, assignments, attendance, live quizzes, success center.
Support admissions pipelineApplicants need public course discovery; admins need review and enrollment handoff.Public course/application routes and admin Applications page.
Enable analytics and supportAt-risk students need early intervention.Success center and class/attendance/performance reports.

Functional Goals

  • User registration, login, logout, refresh token rotation, and current user lookup.
  • Role-based portal navigation for admin, instructor, and student.
  • User, student, instructor, course, class, enrollment, assignment, submission, attendance, document, notification, CMS, quiz, application, report, search, and import/export workflows.
  • Public course browsing and public course application submission.
  • Realtime notifications and live quiz updates using Socket.IO.
  • AI-style Copilot chat, drafts, documents, citations, and audit access.

Overall Description

Product Perspective

EduCore is a monorepo with a React/Vite frontend and an Express/TypeScript backend. MySQL stores relational academic records. MongoDB stores flexible content, notifications, documents, quiz sessions, applications, and Copilot artifacts.

User Classes

Public ApplicantStudentInstructorAdminMaintainer

Constraints

Node.js, Express, React, TypeScript, MySQL, MongoDB, local uploads, JWT auth, Docker Compose service model, and browser-based UI.

Assumptions

Production deployment will replace development secrets, configure SMTP/object storage, enable HTTPS, add CI/CD and backup jobs, and define actual budget and service-level targets.

Architecture Diagram

Deployment Architecture

The repository includes Docker Compose services for MySQL, MongoDB, backend, and frontend. The frontend container serves the built React app through Nginx on port 5173. The backend container exposes port 4000 and depends on database services. Local development also supports XAMPP MySQL and a local MongoDB instance.

Architecture Decisions

DecisionReasonTradeoff
MySQL plus MongoDBRelational records need constraints; content, applications, quiz and Copilot objects benefit from flexible schemas.Two database engines increase backup and operational complexity.
JWT access plus refresh token tableStateless access tokens with revocable refresh sessions.Token storage must be hardened for production.
Socket.IOSimple realtime events for notifications, submissions, and quiz updates.Production scaling needs sticky sessions or adapter/backplane.
Local uploadsSimple MVP implementation for files.Production should use object storage, antivirus scanning, and signed access.
Modular Express routesFeature areas are discoverable and independently maintainable.Service layer separation is partial; large route files should be decomposed later.

Security Documentation

Authentication

EduCore uses JWT access tokens signed with JWT_ACCESS_SECRET and refresh tokens generated with cryptographic random bytes. Refresh tokens are stored as SHA-256 hashes in MySQL and rotated on refresh. Passwords are hashed with bcrypt cost 12.

Authorization

RolePrimary Permissions
AdminFull operational management, users, admissions, courses/classes, documents, CMS, reports, imports/exports, certificate template and revocation, Copilot audit.
InstructorTeaching dashboard, assigned students/classes, Class Studio, assignments, grading, attendance, quizzes, notifications, success center, reports by role.
StudentStudent home, lessons, work submissions, quizzes, documents, gradebook/certificates, inbox, recommendations, Copilot.
PublicPublic course list and course application submission only.

Security Controls Present

  • Helmet: default HTTP security headers.
  • CORS: restricted to configured CLIENT_URL.
  • Rate limiting: 240 requests per minute per limiter context in app.ts.
  • Input validation: Zod validation middleware for many write endpoints.
  • Password hashing: bcrypt hashes, never plain text storage.
  • Refresh token hashing: database stores token hashes, not raw refresh tokens.
  • Upload file size limit: Multer capped at 10 MB and sanitizes filenames.
  • Database constraints: unique email, student code, enrollment, submission, and token hash constraints.
  • Error handling: structured validation and application errors, duplicate handling, generic 500 message.

Security Review Findings

RiskSeverityObservationMitigation
Exposed secret in repository noteHighA probable external API key appears in a plain text project note. The key is not reproduced here.Immediately rotate/revoke the key, remove it from repository history, add secret scanning, and store secrets only in environment variables.
Development JWT defaultsHighEnvironment schema includes development fallback JWT secrets.Fail startup in production if strong secrets are missing; never use defaults in production.
Token storage in localStorageMediumFrontend stores access and refresh tokens in localStorage.Move refresh token to HttpOnly Secure SameSite cookie or harden XSS protections and token lifetime.
Upload validationMediumFile uploads limit size and sanitize names, but no MIME allowlist/virus scanning is documented.Add MIME/content validation, antivirus scanning, object storage, and signed file access.
Audit coverageMediumActivity logs exist, but not every write path is guaranteed to create audit entries.Centralize audit logging for sensitive actions: login, role changes, grade changes, document review, certificate issue/revoke.
Swagger schema completenessLowOpenAPI currently lists many endpoints but has limited request/response schemas.Expand OpenAPI components and examples for all endpoints.
Transport securityHigh in productionLocal URLs use HTTP for development.Use HTTPS/TLS in staging and production; secure cookies and HSTS.

Security Requirements For Production

  • Enable dependency audit and secret scanning in CI.
  • Configure strong secrets, SMTP credentials, and AI provider keys only in secure environment storage.
  • Use HTTPS and secure cookie/session strategy.
  • Limit admin access by least privilege and enforce strong password policy or SSO/MFA.
  • Encrypt backups and restrict file/object storage access.
  • Run penetration testing for auth, RBAC, file upload, IDOR, XSS, CSRF, and injection risk.

2. Create MySQL schema:
   mysql -h 127.0.0.1 -P 3306 -u root < backend/database/schema.sql
3. Install dependencies:
   npm install
4. Seed demo data:
   npm run seed
5. Run full stack:
   npm run dev
6. Open frontend:
   http://localhost:5173
7. Open API docs:
   http://localhost:4000/api/docs

1. Create backend/.env from .env.example.
2. Replace all production secrets and database credentials.

   http://localhost:4000/api/docs

  • MySQL: nightly logical dump, transaction-safe backup where available, 30-day retention, encrypted storage.

  • Uploads: object storage versioning or filesystem snapshot; include submissions, documents, CMS/Copilot assets.
  • Restore tests: monthly restore drill into staging environment.
